Safe Place API

On Restapi startup several actions are taken
Restapi searches for the folder /etc/boxio-restapi-server, if it does not exist then it’s created. In this folder Restapi stores the information that has to persist through the boxio power cycles.
Restapi searches for /etc/boxio-restapi-server/remoteserver.json, if it’s not presents then it’s created with this default configuration

{
  ip: string - default "",
  token: string - default "",
}

The ip attribute holds the ip or the domain name of the server to which the websocket connection has to be created and the token attribute is the token used to connect with the server.
If the ip field is not empty when Restapi is started then the connection is created, otherwise no connection is established.
If there’s no ip the token is ignored, if just the ip is set then the token is requested again to the server and the websocket connection is created.
If both fields are filled, the connection is created using the provided data.

Restapi searches in /etc/boxio-restapi-server/ for the structure.json and automatism.json files, those contain the json configuration: the in the first one we store the endpoints structure and in the second one the wanted behavior of the endpoints.
If Restapi cannot find one or both those files it creates them with an empty json object as content.

Restapi searches for /etc/boxio-restapi-server/aclPermissionMessage.json, if it’s not presents then it’s created with this default configuration

{
  permission_revision: string - default "",
  uuid: string - default "",
  permissions: object - default {}
}

In this version of boxio there’s an mqtt broker running that needs to who the access control list.
We will see that Restapi receives the acl from the server and stores them in a database available to the broker.
Restapi saves the last acl message received in the aclPermissionMessage file.

Since there are other services, such as else, that needs to connect to the broker, Restapi has to store the mqtt credentials in /etc/boxio-restapi-server/mqttCredentials.json.
If this file does not exists it’s created with those default values

{
  enabled: bool - default false,
  url: string - default "",
  port: int - default 0,
  ssl_enabled: bool - default false,
  ssl_verify_certificate: bool - default false,
  cloud_type: string - default "",
  username: string - default "",
  password: string - default "",
  client_id: string - default ""
}

Restapi, using systemctl, enables the dnsmasq and hostapd services.
Then Restapi start listening on port 8443

We said that the mqtt broker on the boxio needs to know the acl and the list of users that have the permission to view and manage the boxio data.
Those informations are stored in an sqlite database managed by this controller.
The database file is /etc/boxio-restapi-server/access-control.sql, if this file is not found when restapi starts then it’s created and initialized.
When the database is created we add to the user table the configurator user with password “configurator” and in the boxio table we add the boxio itself with a random password.
Then the mqttCredentials.json file is updated with the credentials of the boxio.

In order to make the first BOXIO configuration, it's mandatory to invoke the /newborn API.
The system could work in two diffirent modes

  • Connected to a server
  • Standalone
If the mode is standalone, then the rest api interface provides the /auth to make the user able to get an access token. If the Boxio is connect to a server this route will be not available and the authentication has to be performed on the given server
